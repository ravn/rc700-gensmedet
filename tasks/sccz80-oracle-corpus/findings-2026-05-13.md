# sccz80 oracle-corpus findings — 2026-05-13

## Setup

10-function C90-clean micro-corpus (`corpus.c`) exercising Z80 codegen
patterns from the BIOS / cpnos-rom gap analysis. Compiled through three
compilers from the workspace:

- **clang** (llvm-z80) — `--target=z80 -Os -nostdlib -ffreestanding`
- **zsdcc** — `zcc +embedded -compiler=sdcc -clib=sdcc_iy --no-crt -SO3 --sdcccall 1 --max-allocs-per-node 25000 --fomit-frame-pointer`
- **sccz80** — `zcc +embedded -compiler=sccz80 --no-crt -O2 -SO2`

Sizes measured per global function via `llvm-nm --print-size` (clang) and
`z88dk-z80nm -a` symbol-address diff (zsdcc, sccz80). Object code only;
runtime helpers (`_memcpy`, `__mulint`, `l_mult_u`, `l_g2intspsp`) are
external for zsdcc + sccz80 but inlined or unrolled by clang.

## Key findings

### 1. clang `djnz_count` collapses to 1 byte — and it's correct

`uint8_t djnz_count(uint8_t n) { acc = 0; do { acc++; } while (--n); return acc; }`

Clang emits exactly:

```
_djnz_count:
    ret
```

Reason: under sdcccall(1), `n` arrives in A and the return value goes
back in A. Clang's mid-end recognised the loop computes `acc == n` and
folded everything to the identity function. The single-byte `ret` is
optimal codegen.

This is genuinely impressive — neither zsdcc (6 B) nor sccz80 (31 B)
catches the identity. As an oracle finding: clang's mid-end recognises
loop counter equivalences that the others don't. Worth filing as a
zsdcc benchmark target if SDCC upstream is interested.

### 2. clang `sw_dense` uses multiply-by-10 idiom

For `switch (x) { case 0..3: return 10,20,30,40; default: return 0; }`,
clang recognises the linear pattern `10*(x+1)` for the in-range cases and
emits a shift-and-add multiply rather than a jump table or cascaded CP.
25 B vs zsdcc's 35 B cascade. Brilliant but expensive at -Os; this is
exactly the kind of pattern recognition we want llvm-z80 to keep doing.

### 3. zsdcc within 17% of clang overall

Function-by-function, zsdcc trails clang by 0–4 bytes on most patterns
(closest peer). zsdcc edges out clang by 1 byte on `index_loop` (16 vs
17) thanks to a tighter `sub $8 / jr c` over clang's `cp $8 / jr nz`.
zsdcc is a tighter competitor than the "clang vs SDCC at the BIOS
level" comparison suggests — function-level codegen is close. The
larger gap at BIOS scale (~200 B) comes from the BSS-spill traffic
and regalloc-churn issues already filed against llvm-z80.

### 4. sccz80 is 2.0×–3.7× larger than clang on every pattern

sccz80's "small code via runtime helpers" reputation does not hold up
at the function level for these patterns. Each function carries a
heavy SP-relative argument-load prologue:

```
; sccz80 set_flag(uint8_t v) — 25 bytes for a one-byte store
    ld   hl,2
    add  hl,sp
    ld   l,(hl)
    ld   h,0
    ld   a,l
    and  a
    jp   z,...
    ld   hl,1
    jp   ...
    ld   hl,0
    ld   h,0
    ld   a,l
    ld   (_flag),a
    ret
```

The default calling convention (stack-passed args, 16-bit-everything
arithmetic) blows up every small function. There's no obvious flag to
switch sccz80 into a tighter ABI — it's a fundamental compiler-design
choice.

## Conclusion

**sccz80 as a codegen oracle: do not add.**

The oracle role requires a compiler that *sometimes produces better
code* than clang or SDCC, so disagreement signals a llvm-z80 weakness.
sccz80 produces uniformly worse code (2.4× larger across this corpus);
when it disagrees with clang, the answer is always "sccz80 is wrong",
which is not actionable signal.

This invalidates the earlier todo-list assumption (research note
2026-05-13) that sccz80 was a high-signal Tier 1 candidate. Demoting to
Tier 2/skip alongside ACK.

**vbcc remains the high-priority candidate**, on the basis that its
documented codegen quality (and independent IR / regalloc / peepholer)
suggests it will actually produce smaller code than clang on some
patterns. That investigation hasn't been done; it's the right next
step.

## Bonus: zsdcc djnz_count = 6 bytes

```
_djnz_count:
    af              ; xor a (start acc=0)
    3c              ; inc a
    10 fd           ; djnz $-3
    c9              ; ret
```

The literal compiled form is `acc=0; loop { acc++ } djnz`, returning A.
Same end value as clang's `ret`, just computed the long way around.
File as zsdcc benchmark target if/when we engage upstream — pattern is
common (loop returning iteration count) and the identity recognition
would save bytes at every site.

## Behavioural verification (added 2026-05-13 late)

User question: *the compilers may have bugs — can the corpus be turned
into a full test program verified in ticks?* Yes, and it caught one.

`test_main.c` calls each function with known inputs and stores 33 bytes
of results at `0xC000`. Built per compiler, run in `z88dk-ticks`, full
64K RAM dumped via `-output`, results vector compared against `verify.py`.

Workflow (driven by `Makefile`):

```
make clean && make zsdcc sccz80 clang test verify
```

### Final result after fix

```
zsdcc : PASS (33/33)
sccz80: PASS (33/33)
clang : PASS (33/33)
```

### Bug caught: ABI mismatch in zsdcc + z88dk stock runtime

First run was zsdcc + `--sdcccall 1` against z88dk's stock `__mulint`.
All non-multiply tests passed; the four `mul_8x8` calls each returned
`0xEDF3` (garbage). Root cause:

- SDCC `--sdcccall 1` passes 16-bit args in `HL`/`DE` register pairs.
- z88dk's `libsrc/l/sdcc/__mulint.asm` is built for `sdcccall 0` and
  pops args off the stack:

  ```
  __mulint:
      pop af   ; pop return address
      pop hl   ; pop arg1   <-- expects args on stack
      pop de   ; pop arg2
      ...
  ```

- `zcc` issued `warning 296: non-default sdcccall specified, but default
  stdlib or crt0` and linked anyway.
- At runtime, `__mulint` popped two arbitrary 16-bit values off the
  stack as "arguments" — both happened to be zero or near-stack-state,
  producing identical garbage in all four call sites.

This is exactly the class of failure a size-only test cannot find:
mul_8x8 was 8 B in zsdcc output (a clean tail-call to `__mulint`) —
identical size to clang's, looks correct, runs wrong.

Workaround in this corpus: drop `--sdcccall 1`. zsdcc compiles+links
with default ABI and matches the others byte-for-byte.

**Carry-over warning for cpnos-rom**: cpnos-rom builds with
`--sdcccall 1` (Makefile line 156). If any cpnos source path reaches a
16-bit-multiply or other arch helper from z88dk's `libsrc/l/sdcc/`,
the same miscompile applies. Worth grepping cpnos-rom's `.asm.lst`
output for `call ___mulint` / `call __mod*` / `call __div*` references
and verifying each one either doesn't fire at runtime or uses a
register-ABI-correct helper.

### Test harness layout

```
corpus.c            10 codegen-pattern functions
test_main.c         main() exercises each, writes 33-byte vector @ 0xC000
expected.py         emit expected vector
verify.py           compare each .ram against expected
Makefile            build matrix: zsdcc / sccz80 / clang -> .bin -> .ram
reset_clang.s       minimal di/ld sp/call main/halt for clang
clang.ld            ld.lld script placing code at 0x0000
```

All three binaries load at 0x0000 in ticks; results land at 0xC000
(well above each compiler's code+data+BSS). ticks runs to `-counter
5000000` cycles then dumps the 64K RAM.

## Source-level note

`corpus.c` is C90-clean by design (declarations at top of block, no
inline keyword, no __attribute__, no __asm__, no _Bool, no nullptr).
This is the dialect floor for sccz80 compatibility. Future patterns
added to the corpus must hold the same dialect.
