# sccz80 oracle-corpus findings — 2026-05-13

## Setup

10-function C90-clean micro-corpus (`corpus.c`) exercising Z80 codegen
patterns from the BIOS / cpnos-rom gap analysis. Each compiler is run
in its **smallest-code mode**:

- **clang** (llvm-z80) — `--target=z80 -Oz -nostdlib -ffreestanding`
  - `-Oz` ("optimise for size, more aggressively than `-Os`"); ~25 B win
    on this corpus vs `-Os` (single-iteration `copy8` loop stays a loop
    instead of unrolling).
- **zsdcc** — `zcc +z80 -compiler=sdcc -clib=sdcc_iy --opt-code-size -SO3 --max-allocs-per-node 25000 --fomit-frame-pointer`
  - `--opt-code-size` switches SDCC's peephole pass from `sdcc_peeph.3`
    to the size-focused `sdcc_peeph_cs.3` and biases other heuristics
    for density.
  - Note: cpnos-rom production *disables* this flag because the
    ___sdcc_enter_ix helper costs amortise poorly across only 2
    IX-frame functions there. For this stand-alone 10-function corpus
    the flag is a win.
- **sccz80** — `zcc +z80 -compiler=sccz80 -O2 -SO2`
  - sccz80 has no separate size/speed knob; `-O2`/`-SO2` are the max
    peephole levels.

ABI: all three use stack-passed arguments (sdcccall 0 for zsdcc). See
the "ABI mismatch" subsection below for why we couldn't use the
register ABI (sdcccall 1) — z88dk's stock runtime helpers are
sdcccall-0 only.

Sizes measured per global function via `llvm-nm --print-size` (clang)
and `z88dk-z80nm -a` symbol-address diff (zsdcc, sccz80). Object code
only; runtime helpers (`__mulhi3` / `__mulint` / `l_mult_u`) are
external and counted separately if at all.

## Key findings

### 1. clang `djnz_count` collapses to 1 byte — and it's correct

`uint8_t djnz_count(uint8_t n) { acc = 0; do { acc++; } while (--n); return acc; }`

Clang emits exactly:

```
_djnz_count:
    ret
```

Reason: under sdcccall(1), `n` arrives in A and the return value goes
back in A. Clang's mid-end recognised the loop computes `acc == n` and
folded everything to the identity function. The single-byte `ret` is
optimal codegen.

This is genuinely impressive — neither zsdcc (6 B) nor sccz80 (31 B)
catches the identity. As an oracle finding: clang's mid-end recognises
loop counter equivalences that the others don't. Worth filing as a
zsdcc benchmark target if SDCC upstream is interested.

### 2. clang `sw_dense` uses multiply-by-10 idiom

For `switch (x) { case 0..3: return 10,20,30,40; default: return 0; }`,
clang recognises the linear pattern `10*(x+1)` for the in-range cases and
emits a shift-and-add multiply rather than a jump table or cascaded CP.
25 B vs zsdcc's 35 B cascade. Brilliant but expensive at -Os; this is
exactly the kind of pattern recognition we want llvm-z80 to keep doing.

### 3. zsdcc ~1.9× larger than clang on this corpus

Earlier version of this doc claimed "zsdcc within 17% of clang" — that
was an artifact of two confounders that have been removed:
1. `--sdcccall 1` (register-passing ABI) — only ~3% of the 17% was
   compiler quality; the rest was just stack vs register arg passing.
   Removed because z88dk's stock `__mulint` requires sdcccall 0 (see
   ABI-mismatch subsection below).
2. `copy8` via `memcpy()` — both zsdcc and clang tail-called the
   `_memcpy` helper, making the per-callsite size identical at 8 B.
   Replaced with a hand-rolled byte loop so it actually measures
   compiler codegen.

With those confounders gone, zsdcc on the bare codegen patterns is
~1.9× clang's size — a much wider gap than the BIOS-level
"clang vs SDCC −203 B" comparison suggests. The BIOS gap is smaller
because: (a) BIOS uses --sdcccall 1, (b) the absolute baseline is
larger so the ratio is dominated by big functions that hide
per-instruction codegen losses.

zsdcc still beats clang on `index_loop` (tied at 16 B) and matches it
closely on `seq_bss` and `fill_buf` — patterns where SDCC's peepholer
recognises the C idiom cleanly.

### 4. sccz80 is 1.3×–6.8× larger than clang on every pattern

sccz80's "small code via runtime helpers" reputation does not hold up
at the function level for these patterns. Each function carries a
heavy SP-relative argument-load prologue:

```
; sccz80 set_flag(uint8_t v) — 25 bytes for a one-byte store
    ld   hl,2
    add  hl,sp
    ld   l,(hl)
    ld   h,0
    ld   a,l
    and  a
    jp   z,...
    ld   hl,1
    jp   ...
    ld   hl,0
    ld   h,0
    ld   a,l
    ld   (_flag),a
    ret
```

Same stack-passed ABI as zsdcc here, but sccz80 lacks the peephole
passes that compress this. There is no `--opt-code-size` for sccz80;
`-O2`/`-SO2` are already the most aggressive settings. sccz80's
codegen design favours simple/predictable output over density.

## Conclusion

**sccz80 as a codegen oracle: do not add.**

The oracle role requires a compiler that *sometimes produces better
code* than clang or SDCC, so disagreement signals a llvm-z80 weakness.
sccz80 produces uniformly worse code (2.4× larger across this corpus);
when it disagrees with clang, the answer is always "sccz80 is wrong",
which is not actionable signal.

This invalidates the earlier todo-list assumption (research note
2026-05-13) that sccz80 was a high-signal Tier 1 candidate. Demoting to
Tier 2/skip alongside ACK.

## Bonus: zsdcc djnz_count = 6 bytes

```
_djnz_count:
    af              ; xor a (start acc=0)
    3c              ; inc a
    10 fd           ; djnz $-3
    c9              ; ret
```

The literal compiled form is `acc=0; loop { acc++ } djnz`, returning A.
Same end value as clang's `ret`, just computed the long way around.

common (loop returning iteration count) and the identity recognition
would save bytes at every site.

### Bug caught: ABI mismatch in zsdcc + z88dk stock runtime

First run was zsdcc + `--sdcccall 1` against z88dk's stock `__mulint`.
All non-multiply tests passed; the four `mul_8x8` calls each returned
`0xEDF3` (garbage). Then mul_8x8 was swapped out for `mod_10` /
`mod_7` (which the BIOS clang build actually uses) and the *same
class* of bug reappeared: `__moduchar` is also stack-args-only. Every
helper in `libsrc/l/sdcc/` is built for sdcccall 0. Root cause:

- SDCC `--sdcccall 1` passes 16-bit args in `HL`/`DE` register pairs.
- z88dk's `libsrc/l/sdcc/__mulint.asm` is built for `sdcccall 0` and
  pops args off the stack:

  ```
  __mulint:
      pop af   ; pop return address
      pop hl   ; pop arg1   <-- expects args on stack
      pop de   ; pop arg2
      ...
  ```

- `zcc` issued `warning 296: non-default sdcccall specified, but default
  stdlib or crt0` and linked anyway.
- At runtime, `__mulint` popped two arbitrary 16-bit values off the
  stack as "arguments" — both happened to be zero or near-stack-state,
  producing identical garbage in all four call sites.

This is exactly the class of failure a size-only test cannot find:
mul_8x8 was 8 B in zsdcc output (a clean tail-call to `__mulint`) —
identical size to clang's, looks correct, runs wrong.

Workaround in this corpus: drop `--sdcccall 1`. zsdcc compiles+links
with default ABI and matches the others byte-for-byte.

**Carry-over warning for cpnos-rom**: cpnos-rom builds with
`--sdcccall 1` (Makefile line 156). If any cpnos source path reaches a
16-bit-multiply or other arch helper from z88dk's `libsrc/l/sdcc/`,
the same miscompile applies. Worth grepping cpnos-rom's `.asm.lst`
output for `call ___mulint` / `call __mod*` / `call __div*` references
and verifying each one either doesn't fire at runtime or uses a
register-ABI-correct helper.

## Why is clang better? (added 2026-05-14)

Disassembled each compiler's output for the wide-gap functions to
characterise the sources of clang's 1.9× advantage over zsdcc. Three
distinct mechanisms, additive:

### Mechanism 1 — IX-frame overhead (structural, ~60 B / 50% of the gap)

10 of 11 zsdcc-compiled functions begin with `call ___sdcc_enter_ix`
and end with `pop ix; ret`. Each arg access is `ld a, (ix+4)` (3 B)
instead of register-resident `ld a, ...` (1 B from A directly).

Per-function tax: ~6 B prologue/epilogue + ~2 B per arg load.
At 10 functions × ~6 B = ~60 B before counting per-arg overhead.

Clang under llvm-z80 emits leaf functions with **no prologue** at all
when the C language standard permits — args arrive in A / HL and stay
in registers through the body. Example: `djnz_count` is literally
`ret` (1 B).

Why zsdcc can't avoid this:
- z88dk's stock SDCC runtime helpers (`__mulint`, `__moduchar`,
  `__divsint`, etc.) are stack-passed-args only.
- `--sdcccall 1` would give register-passing for **user** functions,
  but the helpers stay stack-based → ABI mismatch on any helper call.
- Result: you can only safely use sdcccall 0 with z88dk's stdlib, and
  sdcccall 0 means IX-frame for arg access.

The cpnos-rom production build uses `--sdcccall 1` and gets away with
it only because no source path reaches a helper. Not a portable
mitigation.

### Mechanism 2 — mid-end pattern recognition (~40–50 B)

LLVM's optimiser recognises high-level C patterns that SDCC's simpler
middle-end doesn't. Concrete instances on this corpus:

#### `djnz_count` — identity recognition (−13 B)

```
clang -Oz:                         zsdcc:
_djnz_count:                       _djnz_count:                  ; 14 B total
    ret                ; 1 B          call ___sdcc_enter_ix
                                      ld l, $00              ; acc = 0
                                      ld b, (ix+4)           ; n in B
                                  l_loop:
                                      inc l                  ; ++acc
                                      djnz l_loop            ; --B, jr nz
                                      pop ix
                                      ret
```

The function semantically computes `acc == n`, and under sdcccall(1)
both arg-in and return-out land in A. Clang's mid-end folds the loop
to identity. SDCC's middle-end faithfully emits the loop.

#### `sw_dense` — multiply-by-10 reduction (−24 B)

```
zsdcc: cascade of 4 × (CP + JR Z + load) for cases 0..3 + default
   → 49 B of pure linear decision tree.
clang: recognises case values are 10·(x+1) for x<4 → multiply pattern
   → 25 B of bit-by-bit shift+add (8-iteration DJNZ multiplier).
```

Both are correct; clang's chooses a tighter representation by
understanding the relationship between case values.

#### `set_flag` and `is_ff` — branchless boolify idiom (−9 B each)

For `(x == K) ? 1 : 0`, clang emits:

```
sub $K          ; A = A - K
add $ff         ; carry out iff A != 0
sbc a, a        ; A = 0xFF if A != 0 else 0
and $1          ; -> 1 or 0
```

This branchless 6-byte sequence is a classic Z80 idiom. zsdcc emits a
branch and two literal loads (`ld a, 1` / `ld a, 0`) costing ~9 more
bytes per site.

### Mechanism 3 — peephole tricks (~15–20 B)

#### `fill_buf` — LDIR-overlap fill (−2 B)

```
clang:                            zsdcc:
    ld de, $1CC                       ld c, 0
    ld a, $AA                     loop:
    ld hl, $1CB                       ld b, 0
    ld (hl), a    ; first byte        ld hl, _bss_buf
    ld bc, $8                         add hl, bc
    ldir          ; copies 8 B        ld (hl), $AA
    ret                               inc c / cmp / jr c, loop
                                      ret
```

Clang recognises "fill N bytes with constant" → uses the overlapping
LDIR trick (store first byte then LDIR from src=dst to dst=dst+1).
SDCC emits a literal loop.

#### `test_bit3` — shift-and-AND vs BIT+branch (−12 B)

```
clang:                            zsdcc:
    rrca                              call ___sdcc_enter_ix
    rrca                              bit 3, (ix+4)
    rrca                              jr z, zero
    and $1                            ld l, 1
    ret                               jr done
                                  zero:
                                      ld l, 0
                                  done:
                                      pop ix
                                      ret
```

Both could use Z80's `BIT n,A` (2 B). zsdcc does use it, but then has
to materialise the result via a branch + two literal loads. Clang
skips the flag entirely with 3× RRCA (bit 3 → bit 0) + AND — same
information, no branch.

### Quantitative breakdown

| Source | Estimated B clang saves |
|---|---:|
| IX-frame elimination across 10 functions | ~60 |
| Pattern recognition (djnz / sw / boolify×2) | ~40-50 |
| Peephole tricks (fill / bit / minor) | ~15-20 |
| **Estimated total** | **~115-130** |
| **Actual gap** (zsdcc 237 − clang 122) | **115** |

Arithmetic confirms the breakdown. About half of clang's advantage is
the structural choice not to pay an IX frame; the other half is the
LLVM middle-end being a more capable optimiser than SDCC's.

### Implications for llvm-z80 work

This corpus suggests where llvm-z80 already leads SDCC: mid-end
optimisations (identity / multiply / boolify) and peephole-level
Z80 idioms (LDIR-overlap fill, RRCA chains). Closing the BIOS-level
gap further likely needs work on **BSS-spill traffic and regalloc
churn** (already filed) rather than on these mid-end patterns —
they're already strong.

## 2026-05-14 update — sdcccall-1 fix + volatile globals

Two follow-up investigations changed the numbers materially.

### Fix: `--sdcccall 1` works with a 19-byte shim helper

The earlier ABI-mismatch trap (z88dk's `__moduchar` is stack-args-only,
incompatible with `--sdcccall 1`) is fixable. `moduchar_sdcccall1.asm`
in this directory provides a 19-byte hand-written `__moduchar` with
SDCC's sdcccall-1 register ABI (A=dividend, L=divisor, returns E /
HL=remainder). Linked ahead of z88dk's stdlib, the linker resolves
calls to our version.

Same approach scales to other helpers if/when needed:

| Helper | Source impact | Effort |
|---|---|---|
| `__moduchar` | `x % N` (8-bit unsigned) | DONE — 19 B asm |
| `__divuchar` | `x / N` (8-bit unsigned) | similar, ~20 B |
| `__mulint`   | 16-bit multiply | ~40 B (shift+add) |
| `__modsint` / `__moduint` etc. | larger types | ~30-50 B each |

Production audit (2026-05-13) said:
- cpnos-rom: NO helper calls in production → no shim needed.
- rcbios BIOS clang build: `___umodqi3` only → shim'd here would
  unblock parallel `--sdcccall 1` SDCC build of the same source.
- rcbios BIOS sdcc build: NO helper calls → already safe with current
  flags.

Recommendation: keep `moduchar_sdcccall1.asm` here in the corpus dir as
reference implementation. Production code that wants to enable
`--sdcccall 1` and uses any helper should copy + adapt the relevant
shim(s). Total project-wide effort if every helper needs one: ~5-8
files, ~150 lines of asm. Bounded and one-time.

### Globals marked volatile to match production

`bss_buf` and `flag` are now `volatile uint8_t`. In real cpnos-rom /
BIOS code these would be ISR-shared or hardware-state and must be
volatile. Adding it here keeps the corpus representative — and
captures the actual production codegen cost of volatile, per memory
rule [[feedback_volatile_blocks_loop_idiom]].

## 2026-05-14 late — reframed via z88dk-native config + dual volatile/non-volatile

User pointed out that z88dk compilers are configured through `zcc`'s
documented options (`-clib`, `-compiler`, `--opt-code-size`, `+target`),
not by tunneling component flags via `-Cs`. Investigation of the
production cpnos-rom + rcbios-in-c/sdcc Makefiles confirmed how
`--sdcccall 1` is "made to work" in production:

```
-Cs"--sdcccall 1" -Cs"--disable-warning 296"
```

That's it. The ABI mismatch (z88dk's helpers are sdcccall-0 only) is
NOT fixed — production simply:

1. Suppresses zcc's warning 296 ("non-default sdcccall specified, but
   default stdlib or crt0").
2. Writes source that **never reaches** a stack-args helper. cpnos-rom
   has zero `__mul/__div/__mod` calls in any built .asm/.lis;
   rcbios-in-c/sdcc the same. Only rcbios-in-c/clang uses one
   helper (`___umodqi3`), and that's from clang's own compiler-rt
   which IS register-ABI-compatible.

So "production sdcccall 1" = workaround by avoidance. The earlier shim
approach (moduchar_sdcccall1.asm, commit 2213e5b) was a real fix but
also reached past z88dk's frontend; reverted.

Corpus revised to match production exactly:
- zsdcc flags: `--opt-code-size -SO3 -Cs"--sdcccall 1" -Cs"--disable-warning 296"`
- Dropped `mod_10` / `mod_7` (helper-bearing) — production never has these.
- Added `mod_8` (power-of-2, lowers to `and 7` inline — no helper).
- Builds both `USE_VOLATILE` and non-volatile binaries per compiler
  (6 total). Both are interesting: non-volatile shows best-case
  codegen, volatile shows the real production cost for ISR-shared /
  hardware-state globals.

### Volatile effects per compiler

- **clang**: +1 B with volatile. Source: `fill_buf` 15→16. Clang's
  LoopIdiomRecognize blocks LDIR-overlap fill when stores are volatile,
  so the 8-byte fill stays a literal byte loop. Matches the documented
  rule [[feedback_volatile_blocks_loop_idiom]].

- **zsdcc**: **−4 B** with volatile. Source: `seq_bss` 17→13. SDCC
  emits two 16-bit `ld hl, NN; ld (addr), hl` stores (4+4+3+3 B = wait
  let me re-count from the asm):

  ```
  ; zsdcc seq_bss under VOLATILE (13 B):
  _seq_bss:
      ld hl, $2211
      ld (_bss_buf), hl
      ld hl, $4433
      ld (_bss_buf+2), hl
      ret
  ```

  This packs each adjacent pair into a 16-bit store — but
  *technically* fuses across what the C abstract machine says are
  4 distinct volatile accesses. Plausibly a SDCC volatile-correctness
  issue (each volatile lvalue access should be preserved); worth
  filing upstream if SDCC engagement deepens.

- **sccz80**: **−4 B** with volatile. Source: `seq_bss` 19→15. Same
  store-merging mechanism as zsdcc, same volatile-correctness question.

### Why does volatile *help* SDCC/sccz80 on `seq_bss`?

Without volatile, both z88dk-frontend compilers track the array base
in a register and use byte stores with adjacent offsets — the
peephole doesn't recognise consecutive symbol+N writes as
combinable. With volatile (which forces a fresh address load for
each store), the peephole notices "two adjacent byte stores via the
same loaded address" and merges them to a 16-bit store. Counter-
intuitive: the more restrictive C semantics produce tighter code.

### Updated headline (production-aligned)

- clang `-Oz` (no extra-clang-rt: no helpers needed): **115 B** (116 vol)
- zsdcc production flags: **177 B** (173 vol) — **1.5× clang**
- sccz80 `-O2 -SO2`: **257 B** (253 vol) — **2.2× clang**

These numbers are now reproducible from the cpnos-rom production
Makefile pattern; the corpus matches what we actually ship.

### Conclusion (re-stable)

- sccz80 as oracle: not worth pursuing (>2× clang on every pattern
  this corpus tests; uniformly worse, low signal).
- zsdcc as oracle: marginal value. ~1.5× clang under matching flags;
  loses on structural (IX frame, pointer args) and mid-end pattern
  recognition. Most disagreements are "zsdcc is worse" — informative
  about cpnos-rom production costs but not about clang codegen bugs.
- ACK measured 2026-05-14 (next section): **4.2× clang**, structurally
  weaker than every other compiler because it targets i8080 not Z80.
  See "ACK / i8080 floor" section below; conclusion is the same — not
  a useful oracle, but provides a concrete 1976-ISA floor data point.

## 2026-05-14 — ACK (Amsterdam Compiler Kit)

ACK is Tier 1: actively maintained, and its output runs on a Z80 (the
i8080 ISA it targets is a strict subset of Z80). It's the weakest Tier
1 entry by a wide margin precisely because it stops at the i8080
subset and can't reach for Z80-only instructions:

- No `DJNZ` (8080 has no B-decrement-and-branch)
- No `JR` / `JR cc` (8080 only has 3-byte absolute `JP` / `JP cc`)
- No `EX DE,HL` (8080 has `XCHG` — same opcode 0xEB, same semantics —
  but no `EX AF,AF'` or `EXX`)
- No `IX` / `IY` (no DD/FD prefixes — confirmed: 0 DD/FD/CB/ED prefix
  bytes in the linked .com)
- No `BIT n,r` / `SET n,r` / `RES n,r` (all CB-prefix)
- No `LDIR` / `LDDR` / `CPIR` / `CPDR` / block I/O (ED-prefix)
- No relative jumps at all → branch-heavy code pays 3 B per branch
  instead of 2 B

Build: `bensuperpc/ack` Docker image (linux/amd64 via Rosetta on this
host; no arm64 manifest). Invocation: `ack -mcpm -O corpus.c
test_main.c -o ack.com`. Output is a `.com` at ORG 0x0100, executes
correctly in `z88dk-ticks -pc 0x100`. Behavioural verification
matches expected vector (both volatile and non-volatile variants
PASS, byte-identical to all other compilers).

### Per-function sizes from `anm -n corpus_ack.o` + `asize`

ACK's `.text` is a concatenation of functions in source order; size

### Headline — all four Tier 1 compilers

| Compiler          | non-vol B | vol B | × clang | ISA reach |
|-------------------|----------:|------:|--------:|-----------|
| clang -Oz         |       115 |   116 |   1.00× | full Z80 (DJNZ, JR, BIT, LDIR, IX/IY, EXX) |
| zsdcc prod flags  |       177 |   173 |   1.5×  | full Z80 |
| sccz80 -O2 -SO2   |       257 |   253 |   2.2×  | full Z80 |
| ack -mcpm -O      |       481 |   481 |   4.2×  | i8080 subset only (no DJNZ/JR/BIT/LDIR/IX/IY/EXX) |

### Why ACK is structurally bound to lose here

The i8080 ISA is missing the exact instructions every pattern in this
corpus uses:

- `djnz_count` — must materialise the loop counter in a register,
  decrement explicitly, and use 3-byte absolute `JNZ`. No 2-byte
  `DJNZ` available. 64 B vs clang's 1 B is partly mid-end (clang
  folds to identity), partly ISA (even without identity, 8080's
  loop overhead is +2 B/iteration).
- `test_bit3` — no `BIT 3, A`. Must `ANI 8` + branch. 44 B for what
  Z80 does in 6 B because the boolify idiom (`SUB / ADC / SBC A,A`)
  works the same on 8080 but ACK's codegen doesn't reach for it.
- `fill_buf` — no LDIR. Must emit a literal loop. 56 B for what
  clang does in 15 B via `LDIR` overlap. SDCC also emits a loop
  here but at 17 B; ACK's loop is 3× larger because every iteration
  pays 3-byte absolute jumps instead of 2-byte JR.
- `seq_bss` — even at 26 B, ACK's "consecutive byte stores" is
  ~75% larger than the others because it can't combine adjacent
  byte stores into 16-bit `LD (NN), HL` (8080 has `SHLD NN` which
  is the same instruction, but ACK's codegen doesn't fuse).

ACK has CSE, register allocation, and a peephole pass, but its
machine description for `i80` is sparse — it does not know about
e.g. `XTHL` for stack-arg swap, `SPHL` for fast SP load, or how to
sequence ALU ops to avoid intermediate stores. The peephole works
on EM (the abstract intermediate code), not on the final i8080
output, so post-codegen peephole-level idioms are missed.

### As an oracle: low value

ACK disagrees with clang on every function (4.2× larger on average),
but the disagreement direction is always "ACK doesn't have the
instruction clang used". That tells us nothing about whether clang's
choice was optimal — it's pre-determined by ISA scope. The
disagreement-as-signal model only works when both compilers have
access to the same instruction repertoire.

There is one indirect use: ACK's output is a useful **lower bound on
how bad the i8080 floor is**. If llvm-z80 codegen ever produces output
within ~2× of ACK, that signals we're missing entire instruction
families (LDIR, BIT, DJNZ, JR — confirming a regression).

### "Small" framing in ACK literature refers to the toolchain, not output

The ACK project self-describes as "small, portable, extremely fast, and
extremely flexible" (davidgiven/ack README, tack.sourceforge.net). That
"small" refers to the **compiler binary / toolchain footprint** —
historically ACK fit in 64K/64K on an 8086 — not to generated code size.

For CP/M C compilers the size-leader folklore (per comp.os.cpm threads,
floppysoftware.es) actually points elsewhere: **BDS C** and **Small C**
are the small-system compilers; **HiTech C** is "the best but memory
hungry"; ACK appears mainly in historical Minix toolchain context, not
in CP/M code-density discussions. Worth recording so future surveys
don't expect ACK to be a density contender.

### Conclusion (re-stable, after ACK)

Tier 1 — supported compilers whose output runs on a Z80 — has exactly
four members, all measured here:

- **clang** (llvm-z80): production primary, the codegen we're optimising.
- **zsdcc**: 1.5×, production secondary, marginal oracle value.
- **sccz80**: 2.2×, no oracle value.
- **ack** (`-mcpm`, i8080 subset): 4.2×, weakest Tier 1 entry —
  structurally bound by stopping at the i8080 subset of the Z80 ISA;
  primarily useful as a "no Z80-specific instructions" floor reference.

**No further compiler exists for Tier 1 to add to this corpus.** vbcc
was hallucinated (no Z80 backend). HiTech `zc` is parked (see
[[feedback_check_hitech_park_note]]). The Tier 1 codegen oracle
approach is exhausted.

Future oracle expansion should look at *hand-written assembly*
references (DRI's CP/M sources, classic Z80 hand-tuned code from
Sinclair / Amstrad games) where the disagreement signal is "human
expert wrote it differently" rather than "another compiler chose
differently".
